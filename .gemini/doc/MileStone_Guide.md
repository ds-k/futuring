# Futuring 프로젝트 아키텍처 및 마일스톤 가이드 🚀

이 문서는 단순한 진행 완료율(체크리스트)을 넘어서, **왜 이런 구조를 선택했고 어떻게 작동하는지** 이해하며 개발을 진행하기 위해 작성된 가이드입니다. 

---

## 🎯 마일스톤 1: 뼈대 구축 (Monorepo & 인프라)
우리는 이 프로젝트를 Next.js(프론트엔드)와 NestJS(백엔드)가 공존하는 **Monorepo**로 구성했습니다. 
*   **왜 이렇게 했나요?** 하나의 Git 레포지토리에서 프론트와 백엔드를 동시에 관리하면, 나중에 타입을 공유하거나 PR을 올릴 때 프론트/백엔드 변경 사항을 한 번에 추적하기 쉬워집니다. 과제 제출 시에도 면접관이 코드를 파악하기 매우 편리합니다.
*   **무엇을 했나요?** 루트 폴더와 백엔드 폴더 각각에 알맞은 `.gitignore`를 설정하여 찌꺼기 파일(`node_modules`, `.env`)이 깃허브에 올라가지 않도록 방지했습니다.

---

## 🎯 마일스톤 2: 백엔드 코어 파이프라인 (진행 중)
"미래로 보내는 메시지"를 처리하기 위해, 백엔드는 다음과 같은 **3단계 파이프라인** 구조로 설계되었습니다. 현재 이 3가지 뼈대가 모두 완성되어 서로 연결(DI - 의존성 주입)되었습니다.

### 1. IPFS 분산 저장소 연동 (`IpfsService`)
사용자가 작성한 메시지 원문을 중앙 서버(AWS S3 등)에 저장하면 회사가 망했을 때 데이터가 날아갑니다. 우리는 이를 방지하기 위해 **Pinata API**를 사용하여 탈중앙화 저장소인 IPFS에 텍스트를 올립니다.
*   **어떻게 구현했나요?** `pinata` 패키지의 의존성 문제를 피하고 가장 안정적으로 통신하기 위해, Node.js의 네이티브 `fetch` API를 사용하여 Pinata의 `pinJSONToIPFS` 엔드포인트에 직접 데이터를 쏘도록 구현했습니다. 업로드 성공 시 영구적인 해시 주소인 **CID**를 반환받습니다.

### 2. 메타데이터 DB 저장 (`PrismaService` + `Supabase`)
IPFS에는 메시지 내용만 올라가고, "누가, 누구에게, 언제 보낼 것인지"에 대한 정보는 DB에 저장되어야 합니다. 우리는 무료이면서도 강력한 PostgreSQL 호환 DB인 **Supabase**를 사용합니다.
*   **왜 Prisma를 썼나요?** TypeScript 환경에서 가장 직관적이고 안전하게(Type-safe) DB를 다룰 수 있는 ORM이기 때문입니다. `schema.prisma`에 `Message` 모델을 정의하고, Supabase의 Transaction Pooler(포트 6543)를 통해 병목 없이 DB와 통신하도록 세팅했습니다.

### 3. 메시지 예약 발송 큐 (`QueueService` + `BullMQ`)
이 프로젝트의 핵심은 **"특정 미래 시간(scheduledAt)에 동작해야 한다"**는 것입니다. Node.js의 기본 `setTimeout`은 서버가 재시작되면 날아갑니다. 그래서 우리는 **Redis** 기반의 큐 시스템인 **BullMQ**를 도입했습니다.
*   **어떻게 구현했나요?** `MessagesService`에서 메시지를 생성할 때, `scheduledAt(예약 시간) - Date.now(현재 시간)` 공식을 통해 **밀리초(ms) 단위의 딜레이**를 계산합니다. 그리고 이 딜레이 값을 BullMQ에 넘겨주면, Redis가 이 시간을 정확히 기억하고 있다가 미래의 특정 시점이 되는 순간 Worker(작업자)를 깨워 메시지를 발송하게 됩니다.

---

### 💡 파이프라인 흐름 요약 (`MessagesService.createMessage`)
컨트롤러(`MessagesController`)로 POST 요청이 들어오면 다음 순서로 톱니바퀴가 맞물려 돌아갑니다:
1.  **IPFS 업로드:** 텍스트를 IPFS에 올리고 `CID` 획득
2.  **DB 기록:** 발신자, 수신자, 예약시간, `CID`를 묶어서 Supabase에 `INSERT`
3.  **큐 예약 등록:** DB에 기록된 메시지 ID와 예약시간을 바탕으로 `BullMQ`에 스케줄링 등록 (예: "30일 뒤에 이 메시지 ID를 발송해!")

## 🎨 마일스톤 3: 프론트엔드 UI 구축 (현재 진행 중)
백엔드 API(`POST /messages`)가 완벽하게 준비되었으므로, 사용자가 텍스트를 입력하고 전송할 수 있는 세련된 UI를 구축합니다.

### 1. 디자인 컨셉 (macOS 엽서 스타일)
사용자 경험(UX)을 극대화하기 위해, 마치 macOS 바탕화면에 띄워진 세련된 편지함 같은 느낌을 주도록 디자인합니다.
*   **유리 질감 & 부드러운 그림자:** 딱딱한 웹페이지가 아닌 애플리케이션 창처럼 보이도록 그림자와 둥근 모서리를 적용합니다.
*   **Vanilla CSS:** Tailwind 등 프레임워크에 얽매이지 않고, 순수 CSS 모듈을 사용하여 1픽셀 단위의 디자인 제어권과 깔끔한 구조를 가져갑니다.

### 2. UI - API 매핑 구조
사용자가 UI에서 입력하는 값들은 백엔드의 DTO 구조와 정확히 일치하도록 매핑됩니다:
*   `To (수신자)` ➡️ **Recipient DID**
*   `From (발신자)` ➡️ **Sender DID**
*   `Time Capsule Date` ➡️ **scheduledAt** (미래의 특정 시점)
*   `Message Body` ➡️ **content** (IPFS에 올라갈 엽서 내용)
*   `Send 버튼` ➡️ **API 호출** (백엔드 파이프라인 작동 트리거)

이 과정을 통해 프론트와 백엔드가 한 치의 오차 없이 깔끔하게 연결됩니다!
